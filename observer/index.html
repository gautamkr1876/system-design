Observer Pattern

JavaScript & Frontend Interview Q&A — click any question in the index to jump to its answer.

🧠 Mental Model

Observer = YouTube channel. The subject is the channel; it keeps the list of subscribers itself. When it uploads (state change), it pings every subscriber on the list directly — no middleman. Subscribers can hit unsubscribe (.off / removeEventListener) anytime.

Why it's not Pub/Sub: Pub/Sub uses a broker (event bus) between publisher and subscriber, and neither knows the other exists. Observer is direct — the subject owns the subscriber list.

Conceptual

1.What is the Observer pattern? Name the two roles.

A behavioral design pattern where one object (the Subject / Observable) keeps a list of dependents and notifies them automatically when its state changes.

  • Subject — owns the subscriber list; exposes subscribe, unsubscribe, notify
  • Observer — a function or object with an update method that reacts to the notification
↑ Back to top
2.What problem does it solve? Why not just call methods directly?

Decoupling. The subject doesn't know who its observers are at design time — they register at runtime. Without it, every new reaction means editing the subject (violates open/closed principle).

Same principle as event delegation, applied to objects: one source, many independent reactions, added/removed without touching the source.

3.Observer vs Pub/Sub vs Mediator — one-line differences.
  • Observer: subject owns the subscriber list. Direct, usually synchronous. One-to-many.
  • Pub/Sub: a broker (event bus) sits between publisher and subscriber; neither knows the other. Usually async. Many-to-many.
  • Mediator: a central coordinator that knows all parties and routes messages between them. Many-to-many through a hub. Used for complex UI workflows (wizard, form orchestration).

Memory trigger: Observer = direct line, Pub/Sub = bulletin board, Mediator = air traffic control.

↑ Back to top
4.Is Observer push-based or pull-based?

Default is push — subject sends data to observers when notifying.

Pull = subject only signals "something changed" and each observer queries for the slice it cares about. Useful when different observers want different parts of the state. Zustand's selector subscriptions are pull-style; RxJS supports both via operators.

5.Where does Observer show up natively in browser and Node?

Browser:

  • EventTarget.addEventListener — DOM events
  • MutationObserver, IntersectionObserver, ResizeObserver, PerformanceObserver

Node:

  • EventEmitter (.on, .emit, .off) — used by streams, HTTP server, process

Both are textbook Observer — the subject keeps its own listener list.

Implementation

6.Implement Subject with subscribe / unsubscribe / notify.
class Subject {
  #observers = new Set();

  subscribe(fn) {
    this.#observers.add(fn);
    return () => this.#observers.delete(fn);
  }

  notify(data) {
    for (const fn of [...this.#observers]) fn(data);
  }
}

Key idea: subscribe returns the unsubscribe function. The caller never has to remember the original reference. Snapshot with [...] before iterating to avoid mutation-during-iteration bugs.

7.Build a tiny EventEmitter with on / off / once / emit.
class EventEmitter {
  #handlers = new Map();

  on(event, fn) {
    if (!this.#handlers.has(event))
      this.#handlers.set(event, new Set());
    this.#handlers.get(event).add(fn);
    return () => this.off(event, fn);
  }

  off(event, fn) {
    this.#handlers.get(event)?.delete(fn);
  }

  once(event, fn) {
    const wrap = (...args) => {
      this.off(event, wrap);
      fn(...args);
    };
    return this.on(event, wrap);
  }

  emit(event, ...args) {
    for (const fn of [...this.#handlers.get(event) ?? []])
      fn(...args);
  }
}

FAANG favorite. Note: off(event, wrap) inside once uses the wrapper's identity — not fn — so it actually removes itself.

8.Add namespace / wildcard support.
emit(event, ...args) {
  // 'user:login' fires 'user:login', 'user:*', and '*'
  const parts = event.split(':');
  for (let i = parts.length; i > 0; i--) {
    const exact = parts.slice(0, i).join(':');
    const wild  = exact + ':*';
    for (const fn of [...this.#handlers.get(exact) ?? []])
      fn(event, ...args);
    for (const fn of [...this.#handlers.get(wild)  ?? []])
      fn(event, ...args);
  }
  for (const fn of [...this.#handlers.get('*') ?? []])
    fn(event, ...args);
}

Pass the original event name as the first arg so wildcard handlers know what fired.

9.TypeScript-typed Observer with generics.
type Listener<T> = (data: T) => void;

class Subject<T> {
  private observers = new Set<Listener<T>>();

  subscribe(fn: Listener<T>): () => void {
    this.observers.add(fn);
    return () => this.observers.delete(fn);
  }

  notify(data: T): void {
    for (const fn of [...this.observers]) fn(data);
  }
}

const userEvents = new Subject<{ id: number; name: string }>();
userEvents.subscribe(u => console.log(u.name));  // typed!

For multi-event emitters use a discriminated union or a Record<EventName, Payload> type.

10.Implement a Signal / Observable primitive.
function createSignal(initial) {
  let value = initial;
  const subs = new Set();

  return {
    get: () => value,
    set: (next) => {
      if (Object.is(value, next)) return;  // skip noop
      value = next;
      for (const fn of [...subs]) fn(value);
    },
    subscribe: (fn) => {
      subs.add(fn);
      return () => subs.delete(fn);
    },
  };
}

const count = createSignal(0);
count.subscribe(v => console.log('count is', v));
count.set(1);  // logs "count is 1"

This is the Observer pattern at the heart of Solid/Preact signals. Real implementations add a reactive graph to auto-track which signals a computation reads.

Tricky & Gotchas

11.Dangling subscriptions & memory leaks — causes & fixes.

Cause: observer never removed; its closure pins component/view in memory → GC can't collect.

Fixes:

  • Always return an unsubscribe from subscribe; call it in cleanup
  • In React: useEffect(() => { const off = sub.subscribe(...); return off; }, [])
  • For native EventTargets: addEventListener('x', fn, { signal: abortCtrl.signal }) auto-removes on abort
  • Advanced: WeakRef + FinalizationRegistry for fire-and-forget references

Memory trigger: if you subscribe, you owe an unsubscribe.

12.What if an observer throws inside notify?

The default for…of loop aborts on throw → remaining observers are skipped. One bad listener silences the rest.

notify(data) {
  for (const fn of [...this.#observers]) {
    try { fn(data); }
    catch (e) { queueMicrotask(() => { throw e; }); }
  }
}

Wrap each call. Re-throw via microtask so errors still reach window.onerror but don't break the loop. This is what Node's EventEmitter does NOT do — uncaught throws there really do crash the process.

↑ Back to top
13.Mutation during iteration — subscribe / unsubscribe mid-notify.

If observer A's callback calls subscribe or unsubscribe on the same subject, you'll skip or double-fire others.

Fix: snapshot the list before iterating.

for (const fn of [...this.#observers]) fn(data);

Trade-off: observers added during this notify won't see this event. That's intentional and matches DOM addEventListener semantics.

↑ Back to top
14.Re-entrancy: observer A triggers another emit that calls A again.

Can become infinite recursion or stack overflow. Guards:

  • Queue — if already notifying, push onto a pending queue and drain after
  • Depth counter with a cap (throw or warn at depth > 100)
  • Microtask defer — queueMicrotask(() => this._actuallyNotify(data)) so the current stack unwinds first

RxJS uses a scheduler abstraction to make this configurable.

↑ Back to top
15.Sync vs async notify — defaults and when to switch.

Default is sync: addEventListener, EventEmitter, most hand-rolled subjects call handlers immediately in subscription order.

Go async when:

  • Subject is on a hot path — don't block the emitter on slow handlers
  • Avoiding re-entrancy (microtask defer)
  • Bursty events (debounce / RAF batch — scroll, resize, input)
  • Handlers themselves are async and you need Promise.allSettled

16.Why WeakRef / WeakMap in observer implementations?

To prevent the subject from keeping its observers alive forever. With WeakRef, the subject holds a weak handle — when nothing else references the observer, GC can collect it.

Trade-offs:

  • Non-deterministic — you don't control when GC fires
  • FinalizationRegistry callbacks are best-effort, not guaranteed
  • Hard to test and debug

Most production code prefers explicit unsubscribe + linters that flag missing cleanups.

17.Why does subject.off(() => {...}) never work?

The arrow at the call site is a brand-new function reference — not the one you passed to on. Set/Map use === identity, so deletion silently fails.

// ❌ Never matches
emitter.on('x', () => doThing());
emitter.off('x', () => doThing());

// ✅ Keep the reference
const handler = () => doThing();
emitter.on('x', handler);
emitter.off('x', handler);

// ✅ Better: return unsubscribe from on()
const off = emitter.on('x', () => doThing());
off();

Real-World Scenarios

18.Build a mini state store + useSyncExternalStore.
function createStore(initial) {
  let state = initial;
  const subs = new Set();
  return {
    getState: () => state,
    setState: (next) => {
      state = typeof next === 'function' ? next(state) : next;
      for (const fn of [...subs]) fn();
    },
    subscribe: (fn) => {
      subs.add(fn);
      return () => subs.delete(fn);
    },
  };
}

// In a React component:
const store = createStore({ count: 0 });
function Counter() {
  const state = useSyncExternalStore(store.subscribe, store.getState);
  return <button onClick={() => store.setState(s => ({ count: s.count + 1 }))}>
    {state.count}
  </button>;
}

useSyncExternalStore is React's official bridge to Observer-based stores. It fixes tearing during concurrent rendering — something hand-rolled useState + subscribe can't do.

19.Cross-component event bus in React — Observer or Pub/Sub?

Honest answer: usually neither. Prefer Context, Zustand, Jotai, or React Query for shared state — they integrate with React's render model.

Pub/Sub wins when: truly fire-and-forget broadcasts where no natural "subject" owns the list (toast notifications triggered from anywhere, telemetry events).

Typed mini Pub/Sub:

type Events = { toast: { msg: string }; route: { to: string } };
class TypedBus {
  #h = new Map();
  on<K extends keyof Events>(k: K, fn: (e: Events[K]) => void) { /* ... */ }
  emit<K extends keyof Events>(k: K, e: Events[K]) { /* ... */ }
}

20.RAF-batched ResizeObserver-style wrapper.
class BatchedSubject {
  #observers = new Set();
  #scheduled = false;
  #latest;

  subscribe(fn) {
    this.#observers.add(fn);
    return () => this.#observers.delete(fn);
  }

  notify(data) {
    this.#latest = data;
    if (this.#scheduled) return;
    this.#scheduled = true;
    requestAnimationFrame(() => {
      this.#scheduled = false;
      for (const fn of [...this.#observers]) fn(this.#latest);
    });
  }
}

Coalesces a burst into one frame. This is what native DOM observers do — fire at most once per animation frame, with the latest value.

21.Autosave: editor → network + localStorage + UI status.

Editor is the subject; three independent observers. Isolate failures so a flaky network doesn't kill the local-storage save:

async notify(data) {
  const tasks = [...this.#observers].map(fn =>
    Promise.resolve().then(() => fn(data))
  );
  const results = await Promise.allSettled(tasks);
  results.forEach(r => {
    if (r.status === 'rejected')
      this.emit('save:failed', r.reason);
  });
}

Then a UI-status observer subscribes to save:failed to show "offline, retrying…".

  2. TickerSubject — receives messages, calls notify(tick)
  3. React UI subscribes via useSyncExternalStore → re-renders only the price cells

Performance tricks:

  • RAF-batch ticks — don't render faster than 60fps
  • Use selector subscriptions so each row only re-renders if its symbol changed
  • Disconnect/reconnect logic on the subject, not on each consumer

Follow-ups

23.Now make it async — what changes?

Wrap notify in queueMicrotask (microtask = after current stack, before next paint) or setTimeout(0) (macrotask = after next paint).

notify(data) {
  queueMicrotask(() => {
    for (const fn of [...this.#observers]) fn(data);
  });
}

If observers themselves are async and you need to await them, use Promise.all / Promise.allSettled over the result of each call.

Connect back: same trade-off as Promise vs setTimeout — microtask runs sooner but starves the event loop if you spam them.

↑ Back to top
24.Scale to 10,000 observers — what breaks first?

Notify is O(n). But the loop itself is rarely the bottleneck — usually one heavy observer dominates. Profile first.

Mitigations:

  • Selector subscriptions — observers fire only when their slice changes (Zustand pattern)
  • Bucket by event type — Map<event, Set<fn>> so only relevant observers wake
  • Batch via RAF or microtask — collapse bursts
  • Remove dead observers — leaks compound at this scale; WeakRef + FinalizationRegistry as last resort

25.How do you unit-test a subject?
test('notifies subscribers and respects unsubscribe', () => {
  const sub = new Subject();
  const fn = vi.fn();
  const off = sub.subscribe(fn);

  sub.notify(42);
  expect(fn).toHaveBeenCalledWith(42);

  off();
  sub.notify(43);
  expect(fn).toHaveBeenCalledTimes(1);  // unsubscribed
});

Always test: (1) subscribers fire, (2) order is stable, (3) unsubscribe actually unsubscribes, (4) a throwing observer doesn't kill the rest.

26.Add priorities so some observers fire first.
subscribe(fn, priority = 0) {
  this.#observers.push({ fn, priority });
  this.#observers.sort((a, b) => b.priority - a.priority);
  return () => {
    this.#observers = this.#observers.filter(o => o.fn !== fn);
  };
}

Trade-off: priorities encourage hidden coupling — "observer X must run before Y." Often a design smell; consider splitting into two sequential subjects (before:save → after:save) instead.

27.Could RxJS replace this? When NOT to use RxJS?

RxJS gives you: Observer + operators (map, filter, debounce, switchMap, retry) + first-class cancellation via Subscription.

Reach for it when: you have streams — async sequences that need composition (search-as-you-type, drag/drop, complex form validation).

Skip when:

  • One-shot subscriptions or simple sync events
  • Bundle size matters (RxJS is ~40KB)
  • Team unfamiliar with reactive style — steep learning curve
  • You already have useEffect + AbortController doing the job in 5 lines

Anti-patterns

28.Every DOM event into an app-wide observer — why bad.
  • Hidden control flow — "who fired this?" becomes impossible to trace
  • No type safety — string event names are typo-prone
  • Memory leaks compound — one missed unsubscribe per feature × 50 features
  • Tight coupling masquerading as decoupling — every consumer still depends on event-name strings

Rule: use the smallest scope that works. Local state → Context → scoped store → global bus. Only escalate when the smaller scope genuinely fails.

↑ Back to top
29.Hand-rolling an EventEmitter inside React — what to use instead.

The problem: a hand-rolled emitter bypasses React's render model. State changes don't trigger re-renders; in concurrent rendering you get tearing (some components see old state, others see new).

What to reach for:

  • useState / useReducer — local state
  • Context — shared state, fine for low-frequency updates
  • Zustand / Jotai — observer-based stores done right
  • useSyncExternalStore — bridge to your own external store

Hand-roll only when bridging something React doesn't own: DOM events, WebSocket streams, third-party SDK callbacks.

↑ Back to top
30.Subjects observing subjects — when chains become anti-patterns.

A → notifies → B → notifies → C → notifies → A. Now you have a cycle, surprising re-fires, and stack overflows. Debugging is brutal because logs interleave.

Fix: single source of truth + derived state.

  • Solid/Preact signals auto-derive without manual subscribe chains
  • Zustand selectors derive from the store; no inter-subject chains
  • Same principle as React's rule: don't compute state from state with useEffect

Memory trigger: if you find yourself writing A.subscribe(() => B.notify(...)), you're building a Rube Goldberg machine. Use derivation, not propagation.
