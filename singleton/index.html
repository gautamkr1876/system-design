Conceptual

1.What is the Singleton pattern in JavaScript?

A design pattern that restricts a class/object to a single instance and provides a global access point to it. In JS, typically implemented via modules, closures, or classes with a static instance reference.

↑ Back to top
2.Why is Singleton "natural" in JavaScript?

Because ES modules are cached after first import. Every import returns the same instance — so export default new Service() is automatically a singleton with no extra boilerplate.

↑ Back to top
3.Difference between Singleton and a module-level object literal const obj = {}?

Both give one shared instance, but the Singleton pattern adds:

  • Lazy initialization
  • Controlled access
  • Private state via closures
  • Enforcement that new returns the same instance

Object literal is eagerly created and fully public.

↑ Back to top
4.Is Math a Singleton? window? console?

Yes, all three are singletons.

  • Math — Built-in object created once at runtime. No constructor (new Math() throws). One shared toolbox for all math operations.
  • window — Single global object per browser tab. Every script sees the same window.
  • console — One shared logging object. console === window.console.
↑ Back to top
5.The two-question mental test for "is this a Singleton?"
  1. Is there exactly one of it in my program?
  2. Can everyone access the same one?

If both = yes → Singleton.

6.Name 5 real frontend uses of Singleton.
  • Logger
  • API/HTTP client
  • WebSocket connection
  • Event bus / PubSub
  • Theme manager

Others: Analytics tracker, Router, Redux/Zustand store, IndexedDB wrapper, Feature flags.

7.Name 3 downsides of Singletons in React/Vue apps.
  • Hidden global state → hard to reason about
  • Tight coupling → components depend directly, not via props
  • Testing pain → must mock the module
  • SSR state leaks between requests
  • No re-renders when state changes
↑ Back to top
8.Singleton vs React Context — when to use which?

Singleton: app-wide service that doesn't need to trigger re-renders (logger, API client, analytics).

Context: shared state components must react to (theme, auth user, locale). Causes re-renders.

Best pattern: Singleton holds the logic; Context exposes a reactive slice.

↑ Back to top
9.Are Redux/Zustand stores singletons?

Yes. Redux: one store created and passed via Provider. Zustand: create() returns a hook tied to one store per module — pure module-singleton pattern.

↑ Back to top
10.Does JavaScript need thread-safety for Singletons?

No for the main thread — JS is single-threaded with an event loop. No volatile, no double-checked locking needed.

But Web Workers each have their own JS realm with their own singleton instance. Singletons are per-realm, not truly global.

Implementation

11.Write a Singleton using closure (IIFE). 
const Singleton = (function () {
  let instance;
  return {
    getInstance() {
      if (!instance) instance = { data: [] };
      return instance;
    }
  };
})();

The closure hides instance — can't be accessed or reassigned externally. Lazy by default.

↑ Back to top
12.Write a Singleton using an ES6 class with private static field. 
class Logger {
  static #instance = null;
  constructor() {
    if (Logger.#instance) return Logger.#instance;
    Logger.#instance = this;
    this.logs = [];
  }
}

The trick: constructor returns the existing instance if one exists. new Logger() === new Logger() is true.

↑ Back to top
13.Write a Singleton using an ES Module (the modern way). 
// logger.js
class Logger { /* ... */ }
export default new Logger();

Modules are cached, so every import logger from './logger' gets the same instance. Most idiomatic JS singleton.

↑ Back to top
14.Why use Object.freeze() on a Singleton?

To make it immutable — prevents accidental or malicious mutation. 

const Config = Object.freeze({
  apiUrl: '...',
  timeout: 5000
});
Config.apiUrl = 'hacked'; // silently fails
↑ Back to top
15.What's the limit of Object.freeze()?

It's shallow. Nested objects are still mutable.

const o = Object.freeze({ a: { b: 1 } });
o.a = 2;     // ❌ fails
o.a.b = 99;  // ✅ succeeds!

For full immutability use a deep-freeze utility.

16.Write a lazy Singleton using a getter. 
const Service = {
  _instance: null,
  get instance() {
    if (!this._instance)
      this._instance = new ExpensiveThing();
    return this._instance;
  }
};

Useful when construction is expensive.

17.Write a Singleton WebSocket manager with auto-reconnect. 
class SocketManager {
  static #instance;
  constructor(url) {
    if (SocketManager.#instance)
      return SocketManager.#instance;
    this.url = url;
    this.connect();
    SocketManager.#instance = this;
  }
  connect() {
    this.ws = new WebSocket(this.url);
    this.ws.onclose = () =>
      setTimeout(() => this.connect(), 1000);
  }
  send(msg) { this.ws.send(JSON.stringify(msg)); }
}
export default new SocketManager('wss://...');

18.Write a Singleton Event Bus with on/off/emit/once. 
class EventBus {
  #listeners = new Map();
  on(event, fn) {
    if (!this.#listeners.has(event))
      this.#listeners.set(event, new Set());
    this.#listeners.get(event).add(fn);
  }
  off(event, fn) {
    this.#listeners.get(event)?.delete(fn);
  }
  emit(event, data) {
    this.#listeners.get(event)?.forEach(fn => fn(data));
  }
  once(event, fn) {
    const wrap = d => { fn(d); this.off(event, wrap); };
    this.on(event, wrap);
  }
}
export default new EventBus();

19.Write the "killer" production API client Singleton. 
class ApiClient {
  static #instance = null;
  constructor(baseURL) {
    if (ApiClient.#instance)
      return ApiClient.#instance;
    this.baseURL = baseURL;
    this.token = null;
    ApiClient.#instance = this;
  }
  setToken(token) { this.token = token; }
  async request(path, options = {}) {
    const headers = {
      'Content-Type': 'application/json',
      ...(this.token && {
        Authorization: `Bearer ${this.token}`
      }),
      ...options.headers,
    };
    const res = await fetch(
      `${this.baseURL}${path}`,
      { ...options, headers }
    );
    if (!res.ok) throw new Error(`API ${res.status}`);
    return res.json();
  }
}
export default new ApiClient('https://api.example.com');

Tricky & Real-World

21.How do you share a Singleton across browser tabs?

Each tab is a separate JS realm. Options:

  • localStorage + storage event — simple, persists
  • BroadcastChannel — modern, message-based
  • SharedWorker — true cross-tab singleton
  • Server + WebSocket — for real apps
↑ Back to top
22.Why is module-level Singleton dangerous in Next.js SSR?

Node reuses modules across HTTP requests. If your singleton holds user-specific state (current user, token), request A's data leaks into request B.

Fix: don't store request-scoped state in a singleton; use AsyncLocalStorage or per-request context.

↑ Back to top
23.How does HMR (Hot Module Replacement) affect Singletons?

HMR can re-evaluate a module while the app runs, creating a new singleton instance while old references still point to the old one. Causes bugs only seen in dev. Use module.hot.accept() carefully.

↑ Back to top
24.Can you break an ES Module Singleton?

The pattern itself is solid, but real-world breakers:

  • Bundler misconfiguration — same package in two node_modules → two copies
  • Micro-frontends — each bundles its own copy
  • Classic "two copies of React" error

Fix: webpack aliases, peer dependencies, or window.__MY_SINGLETON__.

↑ Back to top
25.Can DevTools mutate a client-side Singleton?

Yes. Anything on the client is reachable. Never store secrets or trust the integrity of client-side singletons. Sensitive logic must be server-side.

↑ Back to top
26.What happens to a userService Singleton on logout?

Stale data lingers (cached user, tokens, in-flight requests). Must explicitly call reset() on logout. Subscribers/listeners must be cleaned up or you get memory leaks and "ghost" updates.

↑ Back to top
27.Two micro-frontends bundle two copies of a Singleton — how to fix?
  • Module Federation with singleton: true, eager: true
  • Expose on window: window.__MY_LIB__ = window.__MY_LIB__ || new MyLib()
  • Use peer dependencies + shared bundle

jest.mock('./apiClient', () => ({

    request: jest.fn().mockResolvedValue({ id: 1 })
  }

Or refactor to dependency injection — pass the client via Context/props.

29.How do you make Singleton-using code testable in React?

Three techniques:

  1. Inject via Context or props instead of importing directly
  2. Mock the module: jest.mock('./apiClient')
  3. Expose a reset method: apiClient.__resetForTests()

The new Keyword

31.Core mental model: when does new work?

new only works on "blueprints" for creating objects.

  • Class → ✅
  • Constructor function → ✅
  • Plain object with methods → ❌

Quick check: typeof X === "function" → probably can use new.

↑ Back to top
32.What does typeof Math return?

"object" → ❌ can't use new Math()

Math is a plain namespace object, not a constructor.

↑ Back to top
33.What does typeof Array return?

"function" → ✅ can use new Array()

Array is a constructor.

↑ Back to top
34.Which JS built-ins are "blueprints" (use new)?

Array, Object, Date, Map, Set, Promise, Error, RegExp, WebSocket, FormData, XMLHttpRequest

↑ Back to top
35.Which JS built-ins are "toolboxes" (don't use new)?

Math, JSON, console, localStorage, window, document, Reflect, Intl

These are JS's built-in singletons!

↑ Back to top
36.Can arrow functions be used with new?

No.

const Foo = () => {};
new Foo();  // ❌ TypeError

Arrow functions have no [[Construct]] internal method.

↑ Back to top
37.Can Symbol and BigInt be used with new?

No, despite looking like constructors.

Symbol('id');      // ✅
new Symbol('id');  // ❌ TypeError

They are factory functions — call directly.

↑ Back to top
38.What's the connection between "can't new" and Singletons?

JS's built-in toolboxes (Math, JSON, console) are language-enforced singletons. The inability to new them is the singleton enforcement mechanism. Language designers made them plain objects specifically so you can't create more.

ES Module export:

class MyService { /* ... */ }
export default new MyService();

42.Interview answer: "Is Singleton an anti-pattern?"

"Situational. For stateless infrastructure (loggers, API clients, WebSocket managers) it's pragmatic and clean. For app state, it becomes an anti-pattern — hidden globals, testing pain, bypasses framework reactivity. Modern React/Vue prefer Context, stores, or DI for state."

↑ Back to top
43.Interview answer: "Module vs Singleton pattern?"

"ES modules are implicitly singletons — cached after first evaluation. The Singleton pattern adds explicit control: lazy init, private state via closures, enforcement of single instance. In most JS code, the module is the singleton."

↑ Back to top
44.Why does a class Singleton constructor return the existing instance?

Handles the edge case where someone tries new MyClass() a second time. Without this, you'd get a fresh instance every time. With it: new X() === new X() is always true.

🌟 Deep Dives (Mental Models)

45.Downsides of Singletons in React/Vue — explained with mental model & examples

🧠 Mental Model: A Singleton is a "secret door" in your app. Imagine your component tree as a house with rooms. Normal communication happens through doors — props go down, events bubble up. A Singleton is a secret tunnel that bypasses the floor plan. Any room can teleport to it, grab stuff, change it.

That's why these 5 problems happen:

1. Hidden Global State 🕵️

React's philosophy is "data flows down through props, you can trace it." Singletons break this. 

// ❌ Where does `currentUser` come from?
function ProfilePage() {
  const name = userService.currentUser.name;  // 🕵️ Secret tunnel!
  return <h1>{name}</h1>;
}

A new dev can't tell who sets currentUser by reading the component. They have to search the whole codebase.

Props are a paper trail. Singletons are anonymous tips.

2. Tight Coupling 🔗

The moment you import userService, your component is welded to it. You can't swap it for tests, Storybook, or different contexts. 

// ❌ Married to userService forever
import userService from './userService';
function ProfilePage() {
  return <h1>{userService.currentUser.name}</h1>;
}

// ✅ Works with any user source
function ProfilePage({ user }) {
  return <h1>{user.name}</h1>;
}

Props = LEGO connectors. Singletons = glue.

3. Testing Pain 🧪

Tests should be isolated. Singletons persist between tests, leaking state. 

test('A: login sets user', () => {
  userService.login({ name: 'Alice' });
  expect(userService.currentUser.name).toBe('Alice'); // ✅
});

test('B: no user by default', () => {
  expect(userService.currentUser).toBe(null); // ❌ FAILS — Alice still here!
});

Each test should be a fresh hotel room. Singletons leave the previous guest's stuff there.

4. SSR State Leaks 🏚️ (The Scary One)

In Next.js/SSR, the server reuses modules across requests. Two users can share the same singleton instance. 

// userService.js — looks innocent
class UserService {
  constructor() { this.currentUser = null; }
  login(user) { this.currentUser = user; }
}
export default new UserService();

Timeline of disaster:

  1. Request 1: Alice logs in. userService.currentUser = { name: 'Alice' }
  2. Request 2: Bob hits the server. Server reuses the same module.
  3. Bob's page sees Alice's data! 😱 — a real security bug.

Fix: Use AsyncLocalStorage or per-request context for user-specific data.

In the browser, every user has their own kitchen. On the server, all users share ONE kitchen.

5. No Re-renders 🔕

React re-renders when state/props change. Singletons aren't state — they're just JS objects. Mutating them does nothing visually. 

function App() {
  return (
    <div className={userService.theme}>
      <button onClick={() => userService.toggle()}>
        Toggle Theme
      </button>
    </div>
  );
}
// Click button → singleton.theme flips → screen doesn't update! ❌

React is "asleep" until you call setState. Mutating a singleton is like rearranging furniture while React is asleep.

🎯 The Verdict

Singletons are fine for stateless infrastructure (logger, API client, WebSocket manager). They become problematic when used for stateful app data. For state the UI cares about → use Context or a state library.

46.What is a pattern? What is an anti-pattern? (From the basics)

🧠 Mental Model: Patterns are recipes. Anti-patterns are recipes that look good but ruin the dish.

  • ✅ "Rinse rice first, use 2:1 water ratio" → pattern (proven recipe)
  • ❌ "Add oil to pasta water to cook faster" → anti-pattern (sounds reasonable, ruins the dish later — sauce slides off)

What is a Design Pattern?

A reusable solution to a problem that shows up over and over in software. Thousands of developers faced the same problem ("how do I create one shared logger?"), and the smartest solutions got named and shared.

In 1994, the "Gang of Four" book cataloged 23 common patterns:

  • Singleton — "I need exactly one of this thing"
  • Observer — "When X changes, notify Y and Z"
  • Factory — "Create objects whose type depends on conditions"
  • Strategy — "Same task, swappable algorithms"
  • Decorator — "Add behavior without modifying the class"

Why patterns exist:

  • 🧠 Shared vocabulary — say "it's a Singleton" and another dev instantly gets it
  • 🔁 Battle-tested — someone already hit the bugs and fixed them
  • 🎓 Faster onboarding — new devs recognize patterns

What is an Anti-Pattern?

A "solution" that seems helpful but causes more problems than it solves. Three signs:

  1. It looks like a reasonable solution at first
  2. Many developers reach for it instinctively
  3. It creates worse problems down the line

Famous anti-patterns:

  • God Object — one class that does everything (impossible to test/change)
  • Spaghetti Code — no structure, everything tangled
  • Magic Numbers — if (status === 7) with no context
  • Copy-Paste Programming — bug fix means 20 edits
  • Premature Optimization — over-engineering before there's a real problem
  • Callback Hell — deeply nested callbacks, unreadable

Pattern vs Anti-Pattern

                  Pattern    Anti-Pattern
Looks reasonable?  ✅          ✅
Solves problem?    ✅          ⚠️ Short-term only
Holds up?          ✅          ❌
Recommended?       ✅          ❌

Key insight: The same approach can be a pattern in one context and an anti-pattern in another. Context matters. That's why "Is Singleton an anti-pattern?" is a genuinely interesting interview question.

So is Singleton an anti-pattern?

Singleton is a pattern that's easy to misuse — which is why it gets called an anti-pattern.

It's like sugar. Sugar in moderation is fine. Sugar in everything causes diabetes. Singletons for infrastructure (logger, API client) are fine. Singletons for app state cause maintainability diabetes.

🎯 Interview-Ready Answer

"Singleton is a legitimate Gang of Four pattern — not inherently an anti-pattern. But it's frequently misused, which is why it has that reputation. For stateless infrastructure like a logger or API client, it's pragmatic. For application state, it becomes problematic — hidden globals, testing pain, tight coupling. Modern frameworks like React prefer Context or state libraries for state, and reserve singletons for true infrastructure. The pattern itself isn't the problem; using it for the wrong job is."
